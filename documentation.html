📱 La "app" es una PWA

Una ruta del mismo sitio (/padre) guardada en la pantalla de inicio del celular. Sin App Store, sin segundo codebase, un solo deploy.

✅ Confirmación humana

La API solo transcribe. El párroco siempre ve y aprueba el texto antes de publicar — nada llega al sitio sin su OK.

💰 Costo marginal

Whisper cuesta ~$0.006 USD/minuto: un sermón de 10 min ≈ 6 centavos. Supabase y Vercel en planes gratuitos.

🗄️ Fase 1 — Base de datos: parish_content

Una sola tabla guarda los tres tipos de contenido, diferenciados por la columna section. Script completo en supabase/migrations/001_parish_content.sql.

Modelo de datos

erDiagram
    AUTH_USERS ||--o{ PARISH_CONTENT : "crea (created_by)"

    AUTH_USERS {
        uuid id PK
        text email "cuenta del parroco"
    }

    PARISH_CONTENT {
        uuid id PK "autogenerado"
        text section "sermon | quote | announcement"
        text content "texto transcrito (1 a 10000 chars)"
        boolean published "default true"
        uuid created_by FK "default auth.uid()"
        timestamptz created_at "default now()"
    }

Lógica técnica de cada columna

sectionDiscriminador con CHECK: solo acepta sermon, quote o announcement. Evita 3 tablas idénticas (KISS).
contentEl texto que devolvió Whisper, ya revisado por el párroco. Límite de 10.000 caracteres como protección.
publishedInterruptor de visibilidad: el público solo ve filas con true. Permite "despublicar" sin borrar.
created_bySe rellena solo con auth.uid() (el usuario logueado). Es la base de las políticas de escritura.
created_atEl sitio muestra siempre lo más reciente: el historial nunca se sobreescribe, solo se agregan filas.
Reglas de lectura del sitio:
  • Sermón → la fila más reciente de sermon
  • Frase del día → la fila más reciente de quote
  • Avisos → las últimas 5 filas de announcement

Un índice parcial sobre (section, created_at DESC) WHERE published hace estas consultas instantáneas.

🔐 Fase 1 — Seguridad: Row Level Security

La seguridad no vive en el código de la app sino en la base de datos misma: PostgreSQL evalúa cada consulta contra estas políticas, sin importar desde dónde llegue.

⚠️ Dos pasos manuales en el Dashboard de Supabase (no se pueden hacer por SQL):
  1. Crear el usuario del párroco: Authentication → Users → Add user (email + password, auto-confirm).
  2. Cerrar el registro público: Authentication → Sign In / Up → desactivar "Allow new users to sign up". Así nadie más puede crearse una cuenta y escribir.

🔑 Variables de entorno

Definidas en .env.local (local) y en Vercel → Project → Environment Variables (producción). Plantilla en .env.local.example.

Variable Visibilidad Para qué sirve Dónde obtenerla
NEXT_PUBLIC_SUPABASE_URL Pública URL del proyecto Supabase (BD + Auth) Supabase → Settings → API
NEXT_PUBLIC_SUPABASE_ANON_KEY Pública Key anónima del cliente; la seguridad real la imponen las políticas RLS Supabase → Settings → API
OPENAI_API_KEY 🔒 Server-only Transcripción Whisper. Solo la lee /api/transcribe; jamás con prefijo NEXT_PUBLIC_ platform.openai.com → API keys
NEXT_PUBLIC_SITE_URL Pública URL base para redirects de autenticación localhost:3000 / dominio en producción
